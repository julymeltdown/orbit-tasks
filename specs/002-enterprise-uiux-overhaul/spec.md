# Feature Specification: Orbit Schedule Enterprise UI/UX Re-Architecture

**Feature Branch**: `002-enterprise-uiux-overhaul`  
**Created**: 2026-03-04  
**Status**: Draft  
**Input**: User description: "Orbit Schedule UI/UX 진단 결과를 반영해 객체 중심 IA, 멀티뷰 일관성, 점진적 공개, AI 패널 정비, 접근성/모바일 사용성을 포함한 엔터프라이즈급 UX 재설계"

## Executive Summary

본 기능은 Orbit Schedule의 UI/UX를 페이지 나열형 구조에서 객체 중심 운영 구조로 전환한다.  
핵심 변화는 다음과 같다.

1. 글로벌 내비게이션은 "범위 선택(Scope)"에만 사용한다.
2. 로컬 내비게이션은 "같은 데이터의 표현(View 전환)"에만 사용한다.
3. Work Item, Sprint, Dependency, DSU, Portfolio를 시스템의 일관된 1급 객체로 취급한다.
4. 고급 기능(의존성 관리, AI 분석, 거버넌스)은 점진적으로 공개하여 과부하를 줄인다.
5. AI는 별도 내비게이션 체계가 아닌 컨텍스트 기반 코치로 동작한다.
6. 모바일과 접근성 요구를 동급의 기본 요건으로 반영한다.

본 스펙은 구현 기술을 고정하지 않으며, 사용자 가치와 운영 일관성, 측정 가능한 품질 기준을 정의한다.

## Product Scope

### In Scope

- 글로벌 IA 재편(중복 메뉴 제거, 범위/표현 분리)
- Project Shell 기반 멀티뷰 일관 전환(Board/Table/Timeline/Calendar/Dashboard)
- Work Item 중심 운영 흐름 통합(생성/수정/상태전환/의존성/스프린트 연결)
- Sprint/DSU 운영 루프 재정의
- 협업 흐름(스레드/멘션/인박스) 정비
- AI 코치 패널 단일화 및 설명 가능한 결과 제시
- 포트폴리오 진입 UX 개선(선택형 시작)
- Empty state 표준화
- 모바일/접근성 품질 기준 반영

### Out of Scope

- 영상 회의/콜 기능 자체 개발
- 외부 메신저 전체 대체
- 재무/ERP 결산 기능
- 범용 플러그인 마켓플레이스

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 명확한 내비게이션 계층으로 진입 (Priority: P1)

사용자는 "현재 범위(워크스페이스/프로젝트)"와 "현재 표현(보드/타임라인/테이블)"을 혼동 없이 이해하고 이동하고 싶다.

**Why this priority**: 현재 UX의 가장 큰 실패 원인이 중복 내비게이션과 계층 혼동이므로, 구조 안정화가 전체 개선의 선행 조건이다.

**Independent Test**: 사용자가 3클릭 이내로 원하는 프로젝트에 진입하고, 같은 프로젝트 내에서 여러 뷰를 전환해도 길을 잃지 않으면 독립 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 로그인 후 앱에 진입한 상태, **When** 사용자가 글로벌 내비를 탐색하면, **Then** 글로벌 내비는 범위 선택 항목만 표시한다.
2. **Given** 프로젝트 컨텍스트에 있는 상태, **When** 사용자가 뷰를 바꾸면, **Then** 같은 데이터 집합을 다른 표현으로 전환한다.
3. **Given** 기존에 중복된 메뉴 구조가 있었던 상태, **When** 새 IA를 사용하면, **Then** 동일 목적 항목은 단일 위치에서만 접근 가능하다.

---

### User Story 2 - Work Item 멀티뷰 일관성 확보 (Priority: P1)

팀 구성원은 동일한 Work Item을 보드/테이블/타임라인/캘린더/대시보드에서 일관되게 보고 관리하고 싶다.

**Why this priority**: 제품의 핵심 가치가 "데이터 하나, 여러 관점"이므로 멀티뷰 일관성은 기능 신뢰의 기초다.

**Independent Test**: 하나의 Work Item을 생성/수정했을 때 모든 뷰에서 같은 변경 결과를 확인하면 독립 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** Work Item이 생성된 상태, **When** 사용자가 Board와 Table을 번갈아 보면, **Then** 제목/상태/담당/기한 정보가 동일하게 반영된다.
2. **Given** 일정 필드가 변경된 상태, **When** Timeline과 Calendar를 확인하면, **Then** 날짜 위치가 즉시 동기화된다.
3. **Given** 뷰 전환 전 필터가 적용된 상태, **When** 뷰를 변경하면, **Then** 필터 문맥이 유지된다.

---

### User Story 3 - 보드 조작과 고급 기능 분리 (Priority: P1)

사용자는 칸반 보드에서 빠르게 업무를 이동/편집하고, 의존성 같은 고급 기능은 필요한 순간에만 열어 사용하고 싶다.

**Why this priority**: 일상 업무 흐름에서 과도한 컨트롤 노출은 학습 비용과 실수율을 높인다.

**Independent Test**: 사용자가 보드에서 상태 이동과 기본 편집을 방해 없이 수행하고, 의존성은 별도 모드/상세에서만 편집되면 독립 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 보드 화면에 있는 상태, **When** 사용자가 카드를 드래그하면, **Then** 카드 전체가 일관된 단위로 이동한다.
2. **Given** 일반 보드 사용 중인 상태, **When** 사용자가 화면을 본다면, **Then** 의존성 입력 폼이 기본 툴바를 점유하지 않는다.
3. **Given** 특정 항목 의존성을 수정하려는 상태, **When** 항목 상세 또는 타임라인 의존성 모드를 연다면, **Then** 상/하위 의존성을 함께 확인하고 수정할 수 있다.

---

### User Story 4 - 스프린트와 DSU 실행 루프 정착 (Priority: P1)

스크럼 팀은 스프린트 미설정 상태에서도 다음 행동을 바로 이해하고, 스프린트 운영 중에는 DSU 입력과 블로커 조치를 연결하고 싶다.

**Why this priority**: 스프린트/DSU 루프는 제품의 운영 리듬과 AI 진단 품질을 좌우한다.

**Independent Test**: "스프린트 없음" 상태에서 생성→백로그 연결→DSU 제출→블로커 확인까지 완료되면 독립 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 활성 스프린트가 없는 상태, **When** 스프린트 화면을 열면, **Then** 다음 행동이 포함된 안내형 empty state가 표시된다.
2. **Given** 활성 스프린트가 있는 상태, **When** DSU를 제출하면, **Then** 블로커/요청 신호가 스프린트 문맥에서 확인 가능하다.
3. **Given** 스프린트 항목 필터를 사용하는 상태, **When** 보드로 이동하면, **Then** 스프린트 범위 기반 보기로 연속 작업 가능하다.

---

### User Story 5 - 협업 인박스와 스레드의 역할 분리 (Priority: P1)

사용자는 알림 처리와 대화 생성의 목적을 구분해, 인박스에서는 triage를 하고 스레드는 업무 객체 문맥에서 진행하고 싶다.

**Why this priority**: 인박스/스레드 역할 혼재는 협업 흐름을 느리게 만들고 누락을 발생시킨다.

**Independent Test**: 인박스 탭에서 분류/처리, Work Item에서 스레드 생성/멘션, 딥링크 이동이 자연스럽게 이어지면 독립 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 인박스에 진입한 상태, **When** 사용자가 탭을 보면, **Then** 알림과 요청이 분리된 구조로 표시된다.
2. **Given** Work Item 상세에서 대화를 시작하는 상태, **When** 멘션을 보내면, **Then** 대상자의 인박스에 해당 문맥 링크가 도착한다.
3. **Given** 인박스 항목을 열었을 때, **When** 사용자가 딥링크를 클릭하면, **Then** 관련 객체와 대화 문맥으로 이동한다.

---

### User Story 6 - 포트폴리오 진입 UX 개선 (Priority: P2)

리더는 내부 식별자 입력 없이 포트폴리오를 선택하고, 즉시 상태/리스크/일정 신호를 확인하고 싶다.

**Why this priority**: 포트폴리오 화면은 리더십 대상 핵심 화면이며 첫 진입 마찰이 크면 사용이 중단된다.

**Independent Test**: 포트폴리오 선택 리스트에서 진입해 핵심 KPI를 확인하고 상세 drill-down까지 이동하면 독립 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 포트폴리오 화면 첫 진입 상태, **When** 사용자가 화면을 보면, **Then** 선택 가능한 포트폴리오 목록/최근 항목이 먼저 제시된다.
2. **Given** 특정 포트폴리오를 선택한 상태, **When** 개요를 확인하면, **Then** 리스크와 일정 지표가 요약 제공된다.
3. **Given** 지표를 클릭한 상태, **When** 상세로 진입하면, **Then** 관련 하위 객체 목록으로 연결된다.

---

### User Story 7 - AI 코치의 맥락형 보조 경험 (Priority: P1)

사용자는 AI를 별도 탐색 체계가 아니라 현재 문맥을 해석해주는 코치로 사용하고 싶다.

**Why this priority**: AI는 제품 차별점이지만 내비게이션과 경쟁하면 혼란만 증가한다.

**Independent Test**: 사용자가 현재 컨텍스트에서 AI 패널을 열어 진단 근거와 조치안을 확인하고 실행할 수 있으면 독립 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 프로젝트/스프린트 문맥에 있는 상태, **When** AI 패널을 열면, **Then** 현재 문맥 기준 분석 결과가 표시된다.
2. **Given** AI가 권고를 제시한 상태, **When** 사용자가 근거를 확인하면, **Then** 근거 객체로 이동할 수 있다.
3. **Given** AI 분석이 실패하거나 불확실한 상태, **When** 결과를 보면, **Then** 폴백 결과와 다음 조치 안내가 함께 제공된다.

---

### User Story 8 - 모바일과 접근성 품질 보장 (Priority: P1)

사용자는 모바일에서도 메뉴 접근이 가능하고 가려짐 없는 UI로 주요 작업을 수행하고 싶으며, 키보드/보조기기 사용자도 동일 핵심 흐름을 수행할 수 있어야 한다.

**Why this priority**: 실제 사용환경에서 모바일/접근성 품질 미달은 서비스 사용 불가와 동일한 문제다.

**Independent Test**: 지정 모바일 해상도와 키보드-only 탐색에서 핵심 시나리오를 완료하면 독립 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 모바일 화면에서 앱을 여는 상태, **When** 사용자가 탐색하려면, **Then** 메뉴 진입 컨트롤이 항상 존재한다.
2. **Given** 오버레이/챗 위젯이 열린 상태, **When** 포커스를 이동하면, **Then** 현재 포커스 요소가 가려지지 않는다.
3. **Given** 좁은 화면에서 보드/테이블을 보는 상태, **When** 사용자가 조작하면, **Then** 수평 스크롤과 텍스트/버튼 가시성이 보장된다.

---

### User Story 9 - 관리/거버넌스 화면의 역할 명확화 (Priority: P2)

관리자는 권한, 감사, AI 통제 정책을 별도 관리 영역에서 일관되게 운영하고 싶다.

**Why this priority**: 엔터프라이즈 제품에서 운영통제는 필수이며 일반 업무 화면과 혼재되면 오조작 위험이 커진다.

**Independent Test**: 관리자 계정에서 정책 조회/변경, 일반 사용자 계정에서 차단 동작이 재현되면 독립 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 관리자 권한 사용자 상태, **When** Admin 영역에 진입하면, **Then** 권한/감사/AI 통제 메뉴가 제공된다.
2. **Given** 일반 사용자 상태, **When** 관리자 경로로 접근하면, **Then** 접근이 거부된다.
3. **Given** 정책이 변경된 상태, **When** 관련 기능을 사용하면, **Then** 변경된 정책이 즉시 반영된다.

### Edge Cases

- 사용자가 워크스페이스를 하나도 가지지 않은 경우, 기본 진입 화면은 다음 행동(생성/초대 요청)을 제시해야 한다.
- 사용자가 중간에 권한을 상실한 경우, 현재 화면은 안전하게 읽기 전용 또는 접근 거부 상태로 전환되어야 한다.
- 뷰 전환 중 데이터가 비어 있는 경우에도 동일한 객체 중심 문맥(프로젝트/필터/기간)은 유지되어야 한다.
- 네트워크 지연 또는 부분 실패 상황에서 사용자가 이미 수행한 조작의 상태를 이해할 수 있어야 한다.
- AI 결과가 신뢰 임계치 미만인 경우, 과감한 권고 대신 확인 질문/보수적 조치가 우선 제시되어야 한다.
- 오버레이(패널/챗/모달) 동시 노출 시 핵심 CTA가 가려지지 않아야 하며 포커스 순서가 보존되어야 한다.
- 긴 텍스트(한국어/영어 혼합)에서도 카드/버튼 레이아웃이 깨지지 않아야 한다.
- 드래그 불가 환경(키보드 전용, 터치 제한)에서도 상태 전환 대체 수단이 제공되어야 한다.

## Requirements *(mandatory)*

### Functional Requirements

#### IA and Navigation

- **FR-001**: 시스템은 글로벌 내비게이션을 범위 이동 용도로 제한해야 한다.
- **FR-002**: 시스템은 프로젝트 내부에서만 뷰 전환(Board/Table/Timeline/Calendar/Dashboard)을 제공해야 한다.
- **FR-003**: 시스템은 동일 목적 메뉴가 여러 영역에 중복 노출되지 않도록 해야 한다.
- **FR-004**: 시스템은 현재 범위(워크스페이스/프로젝트)와 현재 뷰를 항상 명시해야 한다.
- **FR-005**: 시스템은 신규 IA에서 핵심 경로를 3단계 이하로 접근 가능하게 해야 한다.

#### Object-Centric Work Model

- **FR-006**: 시스템은 Work Item을 공통 객체로 관리하고 모든 뷰에서 동일 식별자와 상태를 유지해야 한다.
- **FR-007**: 사용자는 Work Item을 생성할 수 있어야 한다.
- **FR-008**: 사용자는 Work Item의 제목/상태/담당/우선순위/일정을 수정할 수 있어야 한다.
- **FR-009**: 사용자는 Work Item을 보존 가능한 방식으로 보관(archive)할 수 있어야 한다.
- **FR-010**: 시스템은 Work Item 변경 이력을 추적 가능하게 기록해야 한다.

#### View Parity and Consistency

- **FR-011**: 시스템은 Board/Table/Timeline/Calendar/Dashboard를 동일 데이터셋의 표현으로 제공해야 한다.
- **FR-012**: 뷰 전환 시 필터/정렬/검색 문맥이 유지되어야 한다.
- **FR-013**: 일정 필드 변경은 Timeline/Calendar에 즉시 반영되어야 한다.
- **FR-014**: 대시보드 지표는 drill-down으로 근거 객체 목록에 연결되어야 한다.
- **FR-015**: 시스템은 저장 가능한 사용자/팀 뷰 구성을 제공해야 한다.

#### Progressive Disclosure

- **FR-016**: 보드 기본 툴바는 고빈도 작업 중심으로 구성되어야 한다.
- **FR-017**: 의존성 관리 기능은 기본 보드 헤더에서 분리되어야 한다.
- **FR-018**: 의존성 편집은 항목 상세 또는 전용 모드에서 수행되어야 한다.
- **FR-019**: 고급 옵션은 기본 UI를 가리지 않는 방식으로 단계적으로 노출되어야 한다.

#### Sprint and DSU

- **FR-020**: 시스템은 활성 스프린트 부재 시 행동 지향 empty state를 제공해야 한다.
- **FR-021**: 사용자는 스프린트를 생성하고 목표/기간/범위를 설정할 수 있어야 한다.
- **FR-022**: 사용자는 스프린트 백로그를 구성할 수 있어야 한다.
- **FR-023**: 사용자는 DSU를 제출하고 팀은 DSU 기록을 조회할 수 있어야 한다.
- **FR-024**: DSU 신호(블로커/요청/진행상태)는 스프린트 문맥에 연결되어야 한다.

#### Collaboration (Thread/Mention/Inbox)

- **FR-025**: 시스템은 Work Item 문맥에서 스레드를 생성할 수 있어야 한다.
- **FR-026**: 시스템은 멘션 이벤트를 인박스 항목으로 전달해야 한다.
- **FR-027**: 인박스는 최소한 알림과 요청을 분리해 표시해야 한다.
- **FR-028**: 인박스 항목은 관련 객체로 이동 가능한 링크를 제공해야 한다.
- **FR-029**: 사용자는 인박스 항목을 처리/해결 상태로 전환할 수 있어야 한다.

#### Portfolio

- **FR-030**: 포트폴리오 첫 진입은 식별자 직접 입력이 아닌 선택 기반 흐름이어야 한다.
- **FR-031**: 시스템은 최근 접근/즐겨찾기 포트폴리오를 표시해야 한다.
- **FR-032**: 포트폴리오 개요는 핵심 일정/리스크 지표를 한 화면에서 제공해야 한다.
- **FR-033**: 개요 지표는 상세 객체 목록으로 이동 가능해야 한다.

#### AI Assistant

- **FR-034**: 시스템은 AI 보조 경험을 단일 주요 표면으로 제공해야 한다.
- **FR-035**: AI 표면은 내비게이션 대체 항목을 포함하지 않아야 한다.
- **FR-036**: AI 결과는 현재 컨텍스트(프로젝트/스프린트/선택 항목)에 기반해야 한다.
- **FR-037**: AI 권고에는 근거 객체 참조를 포함해야 한다.
- **FR-038**: AI 실패/저신뢰 시 시스템은 폴백 결과와 다음 행동을 제시해야 한다.

#### Authentication, Profile, Team, Governance

- **FR-039**: 사용자는 회원가입 후 이메일 확인을 거쳐 로그인할 수 있어야 한다.
- **FR-040**: 최초 로그인 사용자는 프로필 필수 설정 완료 후 제품을 사용해야 한다.
- **FR-041**: 두 번째 로그인부터는 프로필 완료 사용자가 즉시 업무 화면으로 진입해야 한다.
- **FR-042**: 회원가입 완료 시 기본 워크스페이스가 자동 생성되어야 한다.
- **FR-043**: 사용자는 워크스페이스 이름 기반으로 식별/관리할 수 있어야 한다.
- **FR-044**: 관리자는 팀 멤버를 초대하고 역할을 부여할 수 있어야 한다.
- **FR-045**: 관리 기능(권한/감사/AI 통제)은 권한 기반으로 접근 제한되어야 한다.

#### Empty States and Guidance

- **FR-046**: 모든 핵심 빈 상태는 상태 설명, 1개 이상의 주요 CTA, 보조 안내를 제공해야 한다.
- **FR-047**: 빈 상태 CTA는 사용자를 다음 완료 가능한 작업으로 즉시 이동시켜야 한다.

#### Accessibility and Responsive

- **FR-048**: 핵심 사용자 흐름은 키보드만으로 수행 가능해야 한다.
- **FR-049**: 포커스 요소는 패널/위젯/오버레이로 가려지지 않아야 한다.
- **FR-050**: 모바일에서 메뉴 접근 경로는 항상 보장되어야 한다.
- **FR-051**: 모바일 화면에서 주요 버튼/텍스트/카드 정보가 잘리지 않아야 한다.
- **FR-052**: 상태 전달은 색상만이 아니라 텍스트로도 제공되어야 한다.

### Key Entities *(include if feature involves data)*

- **Workspace**: 조직 운영의 최상위 범위. 이름, 역할, 기본 여부, 소속 프로젝트를 가진다.
- **Project**: Work Item과 협업이 수행되는 실행 단위. 여러 뷰와 스프린트를 가진다.
- **View Configuration**: 같은 데이터셋을 어떤 방식(보드/테이블/타임라인/캘린더/대시보드)으로 표현할지 정의하는 사용자/팀 설정.
- **Work Item**: 업무 실행의 기본 객체. 상태, 담당, 우선순위, 일정, 의존성, 대화 문맥과 연결된다.
- **Dependency**: Work Item 간 선후행 관계. 일정 영향과 리스크 판단의 근거가 된다.
- **Sprint**: 기간과 목표를 가진 반복 실행 단위. 백로그와 DSU를 포함한다.
- **Backlog Entry**: 스프린트/프로젝트 실행 큐 항목. 우선순위와 상태를 가진다.
- **DSU Entry**: 일일 실행 보고 객체. 원문과 구조화 신호를 함께 가진다.
- **Thread**: 업무 객체 문맥의 대화 단위. 멘션과 인박스 이벤트를 발생시킨다.
- **Inbox Item**: 사용자가 처리해야 할 알림/요청 항목. 출처 객체와 행동 상태를 가진다.
- **Portfolio**: 다수 프로젝트의 상위 집합. 일정/리스크/용량 관점을 제공한다.
- **AI Evaluation**: 일정 건강도 분석 결과. 권고/질문/근거/신뢰 정보를 포함한다.
- **Policy Control**: 관리자 설정 객체. 권한, 감사, AI 통제 규칙을 담는다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 신규 사용자 90% 이상이 로그인 후 2분 이내에 유효한 작업 화면(프로젝트 뷰)으로 진입한다.
- **SC-002**: 핵심 5개 작업(보드 이동, 항목 생성, 스프린트 생성, DSU 제출, 인박스 처리)의 완료 성공률이 85% 이상이다.
- **SC-003**: 사용성 테스트에서 "내가 어디에 있는지 알겠다" 항목의 긍정 응답이 80% 이상이다.
- **SC-004**: 동일 목적 메뉴 중복 노출 항목 수가 0이어야 한다.
- **SC-005**: Work Item 단일 변경이 5개 뷰 모두에 반영되는 검증 케이스 통과율이 100%다.
- **SC-006**: 활성 스프린트 미설정 사용자 중 70% 이상이 empty state CTA를 통해 스프린트를 생성한다.
- **SC-007**: 인박스 항목에서 관련 객체로 이동하는 deep-link 성공률이 99% 이상이다.
- **SC-008**: AI 분석 요청 중 실패 케이스의 100%에서 대체 결과 또는 다음 행동 안내가 제공된다.
- **SC-009**: 모바일 표준 테스트 해상도에서 가로 오버플로우로 인한 핵심 CTA 가림 결함이 0건이어야 한다.
- **SC-010**: 키보드-only 접근성 점검에서 핵심 흐름 차단 결함이 0건이어야 한다.
- **SC-011**: 포트폴리오 진입에서 식별자 수동 입력 요구 비율이 0%가 되어야 한다.
- **SC-012**: 배포 후 4주 내 관련 UX 혼란성 문의(내비/화면 찾기) 티켓이 기존 대비 40% 이상 감소한다.
